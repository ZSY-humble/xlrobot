# 🎯 XLeRobot 自我遥操作 ACT 采集手册

> **适用场景**：只使用 XLeRobot 本体，一只手推动左臂做示教，
> 程序读取左臂关节并镜像到右臂执行，同步采集右臂关节 + 相机数据，
> 用于训练 ACT / Diffusion / VLA 等模仿学习策略。
>
> **版本**：2026-06-18 · **环境**：`conda activate lerobot`

---

## ✨ 一、能力速览

| 能力 | 状态 | 说明 |
|---|---|---|
| 左臂拖动示教 | ✅ 已实现 | 连接后关闭左臂扭矩，人手推动左臂 |
| 左→右 镜像跟随 | ✅ 已实现 | `act/mirror.py` 负责左右关节映射和取反表 |
| ACT 采集脚本 | ✅ 已实现 | `act/record_self_teleop.py` 自写采集主循环 |
| 边采集边可视化 (Rerun) | ✅ 支持 | 默认开启，可用 `--no-display` 关闭 |
| 数据集 schema | ✅ 固定 | `state=(6,)`、`action=(6,)`，只录右臂 6 关节 |

> 💡 核心约束：这个流程**没有两台外部 SO101 主臂**。
> 左臂只是 XLeRobot 本体的一部分，关闭扭矩后作为可拖动示教臂；
> 右臂是执行臂，也是 ACT 策略最终要控制的对象。

## 🧭 概念澄清（先理清角色）

| 角色 | 作用 | 是否接相机 |
|---|---|---|
| **左臂** | 关扭矩，人手推动，提供示教轨迹 | ❌ 不单独接相机 |
| **右臂** | 根据左臂镜像目标实时跟随，是真正被训练的执行臂 | ✅ 右腕相机装在这里 |
| **桌面相机** | 看全局任务场景 | ✅ 接到 XLeRobot 相机配置里 |

> 🌟 **核心逻辑**：左臂负责“给动作”，右臂负责“执行动作并被记录”。
> 因此 ACT 数据集只保存右臂 state/action，再配合桌面相机和右腕相机。

## 🛠️ 二、采集前准备

### 1. 激活虚拟环境
```bash
conda activate lerobot
```

### 2. 确认 HuggingFace 用户名
```bash
export HF_USER=$(huggingface-cli whoami | head -n 1)
echo $HF_USER
```

### 3. 端口与设备清单
建议用 `udev` 规则给设备起固定别名，避免每次重启端口漂移。

| 角色 | 默认端口 | 备注 |
|---|---|---|
| XLeRobot bus1（左臂 + 头部） | `/dev/ttyACM0` | `FOLLOWER_PORT1` |
| XLeRobot bus2（右臂 + 底盘） | `/dev/ttyACM1` | `FOLLOWER_PORT2` |

本机当前实测：

```bash
export FOLLOWER_PORT1=/dev/ttyACM0
export FOLLOWER_PORT2=/dev/ttyACM1
```

查看端口：
```bash
ls /dev/ttyACM* /dev/ttyUSB* 2>/dev/null
dmesg | grep tty | tail -20
```

### 4. 相机布局（重要 ⚠️）

**相机都挂在 XLeRobot 这台机器上，由 `act/config.py` 统一配置。**

| 相机 | 安装位置 | 作用 | 推荐命名 |
|---|---|---|---|
| 右腕相机 | 右臂末端（夹爪附近） | 抓取近景 | `right_wrist` |
| **桌面相机** | 桌面 / 三脚架 / 桌沿外部 | **全局场景，看任务全貌** | `top` 或 `front` |

> 🌟 **ACT 训练最佳实践**：腕部相机 + 桌面全局相机组合。
> 腕部看"精细抓取动作"，桌面看"物体相对位置"，缺一不可。

#### 检查相机
```bash
ls /dev/video*
python act/check_cameras.py
```

本机当前实测：

```bash
export CAM_TOP=/dev/video3
export CAM_RIGHT_WRIST=/dev/video4
```

#### 区分多路相机的诀窍
```bash
# 拔/插某一路相机前后对比 dmesg，确认 /dev/videoN 对应物理设备
dmesg | tail -5
```

> ⚠️ Linux 下 `index_or_path` 一律建议写 **完整路径** `/dev/video0`，不要写 `0`，避免热插拔后错位。
> 更稳的做法：用 `udev` 规则按 USB 端口给相机起固定别名（如 `/dev/cam_left_wrist`）。

## 🧪 三、第一步：先验证左推右跟

> **⭐ 强烈建议**：正式录数据前，先跑一次自我遥操作验证，
> 确认左臂关扭矩后可拖动、右臂跟随方向正确。

```bash
python act/teleoperate_self.py --no-cameras --no-display --max-relative-target 5.0
```

确认无异常后，再打开相机和 Rerun：

```bash
python act/teleoperate_self.py
```

默认是 delta 变化量跟随，不是把左臂绝对值直接塞给右臂：

```text
right_target = right_start + mirror(left_current - left_start)
```

### ✅ 自检清单
- [ ] 双臂上电、零位无异常抖动
- [ ] 左臂扭矩关闭后可以手推，头部和右臂仍保持上电
- [ ] 左臂动 → 右臂对应关节跟随（**没有反向**）
- [ ] 夹爪开/合行程对得上
- [ ] Rerun 里能看到 `action.right_arm_*.pos` 与 `observation.state.right_arm_*.pos` 趋势一致
- [ ] 30Hz 不掉帧（终端打印的循环频率稳定）

> ❌ 若方向反：先跑 `python -m act.mirror probe`，按结果调整 `NEGATE_JOINTS`。

## 🚀 四、🌟 正式采集命令（核心指令）

### 4.1 完整模板

```bash
python act/record_self_teleop.py \
  --num-episodes 50 \
  --episode-time 20 \
  --reset-time 10 \
  --task "Pick up the red cube and place it into the box"
```

> 💡 这个脚本在 `act/record_self_teleop.py`，显式写死 ACT 数据集 schema：
> `observation.state=(6,)`、`action=(6,)`，只包含右臂 6 关节；
> 左臂只作为拖动示教输入，不写进训练 action。

### 4.2 参数详解（重点记住）

| 参数 | 作用 | 推荐值 |
|---|---|---|
| `--no-display` | 关闭 Rerun 可视化 | 调试 headless 时使用 |
| `--num-episodes` | 采集多少段（episode） | ACT 一般 50–100 |
| `--episode-time` | 单段最长时长 | 抓取类 15–30 s |
| `--reset-time` | 段间复位时间（你摆物体回原位） | 8–15 s |
| `--task` | 任务文本（语言条件策略必需） | 写明任务目标 |
| `--push-to-hub` | 录完上传 HF | 调试期不加 |
| `--root` | 本地数据集目录 | 默认 `$HF_LEROBOT_HOME/<repo-id>` |

## ⌨️ 五、采集时的键盘控制（务必记牢）

> 🌟 这是采集体验的核心：能否快速「成功就保存、失败就重录」直接决定数据质量。

| 按键 | 行为 | 使用时机 |
|---|---|---|
| `→` 右箭头 | **结束本段并保存** | 这一段你完成得不错，立刻保存 |
| `←` 左箭头 | **丢弃本段，重新录** | 这段失败了 / 左臂滑了 / 物体摆错了 |
| `Esc` | **整体停止采集** | 不想录了 / 想中途结束保存已采的 |

### ⚠️ 注意事项
1. **必须在物理机本地终端运行**（SSH 远程终端无法收键盘事件）
2. **Linux 监听键盘可能需要 `sudo`** 或将用户加入 `input` 组
3. **Rerun 窗口在前台不影响**——按键事件由 pynput 全局监听
4. 如果按键无响应，检查是否进入 headless 模式、`DISPLAY` 是否存在，以及 `pynput` / 输入权限是否正常

## 🎬 六、Rerun 边采集边可视化（核心功能）

### 6.1 启用方式
```bash
python act/record_self_teleop.py
# 或关闭显示：
python act/record_self_teleop.py --no-display
```

### 6.2 启动后会自动呈现
- 🎥 **两路相机实时画面**（top 桌面 / right_wrist 右腕）
- 📈 **右臂 action**：`action.right_arm_*.pos`
- 📈 **右臂 observation**：`observation.state.right_arm_*.pos`
- ⏱️ **时间轴**：精确到帧的对齐

### 6.3 边采边看的"质量自检三连"
| 看哪里 | 应该是什么样 | 不对就停 |
|---|---|---|
| 相机画面 | 两路都正常显示，无黑屏、无延迟 | 检查 `/dev/video*` 是否被占 |
| `action` vs `observation` 曲线 | 趋势一致，右臂跟随平滑 | 跟随有迟滞，可能 USB / CPU 瓶颈 |
| 帧率（终端日志） | 稳定 30 Hz | 掉到 25 以下要降相机分辨率 |

### 6.4 Headless 调试
当前 `act/record_self_teleop.py` 走本地 Rerun，远程/headless 场景建议先用：
```bash
python act/record_self_teleop.py --no-display
```

## 🧠 七、标定与镜像表

第一次接入 XLeRobot，或者动过电机零位，需要先做整机标定：

```bash
python act/calibrate_xlerobot.py
```

当前 ACT 链路默认只手动标定 XLeRobot 本体左右臂：

```text
left_arm_*  6 个电机
right_arm_* 6 个电机
```

头部和两轮底盘不参与 ACT 自我遥操作采集，会由脚本保留已有标定或补齐字段。

然后验证左臂到右臂的镜像方向：

```bash
python -m act.mirror show
python -m act.mirror probe
```

> 标定结果保存在
> `~/.cache/huggingface/lerobot/calibration/robots/xlerobot_2wheels/` 下。
> 当前真机实测 delta 跟随时 `DEFAULT_NEGATE_JOINTS = frozenset()`，
> 即默认不额外取反；镜像方向仍可由环境变量 `NEGATE_JOINTS` 临时覆盖。

## ⚠️ 十、常见坑位与排查

| 现象 | 原因 | 解决 |
|---|---|---|
| feature 名对不上，dataset 校验失败 | 混用了旧数据或旧脚本 | 使用 `act/record_self_teleop.py`，脚本显式写 6 维右臂 schema |
| 右臂方向反 / 行程不对 | 标定缺失或镜像取反表不对 | 重新标定，或跑 `python -m act.mirror probe` |
| Rerun 不弹窗 | headless / WSL / 纯 SSH | 本地物理机运行；或加 `--no-display` 先只采数据 |
| 帧率掉到 < 25 fps | 视频编码占 CPU | 降相机分辨率 / `encoder_threads=4` / 减一路相机 |
| `→` 键无反应 | 没有键盘监听权限 | 用本地物理终端 + 检查 `input` 组 / 必要时 `sudo` |
| 右臂跟不上、抖动 | USB 总线带宽不够 / 控制频率过高 | 降相机分辨率，确保两条总线稳定 |
| 录到一半某段不满意 | 操作失误 | **按 `←` 重录**，别按 `→` |
